# AI Chat Exporter Specifications

This document outlines the detailed specifications for the `ai-chat-export` unified CLI wrapper, as well as the underlying `copilot-chat-export`, `opencode-chat-export`, and `claude-code-chat-export` tools. These tools export chat sessions from VSCode Copilot, OpenCode, and Claude Code into Obsidian-compatible Markdown formats.

## 1. High-Level CLI Behavior

All individual CLI tools share a common interface and behavior structure for consistency. The `ai-chat-export` script serves as a unified frontend that delegates commands to the specific exporters.

### Command-Line Arguments & Flags
- `title` (Positional): The session title to search for. Supports partial matches (case-insensitive for Copilot and Claude, `LIKE %query%` for OpenCode). OpenCode and Claude also support searching by exact session ID.
- `--source`, `-s <source>`: (Only in `ai-chat-export`) Filters which exporter to use. Choices are `all` (default), `opencode`, `copilot`, or `claude`.
- `--list`, `-l`: Lists recent sessions instead of exporting.
- `--days`, `-d <int>`: The number of days to look back when listing sessions. Defaults to `7`.
- `--output`, `-o <path>`: Overrides the default output directory and specifies an exact file path for the exported Markdown file.
- `--all`, `-a`: If multiple sessions match the `title` query, this flag automatically exports all of them without prompting.

### Listing Sessions
When invoked with `--list`, the tools calculate a timestamp cutoff based on `--days`. They retrieve sessions newer than this cutoff and output a sorted list (newest first) displaying the timestamp, source tag, session title, and session ID.
When using the `ai-chat-export` wrapper, results from all selected sources are merged into a single list, sorted by timestamp descending, and prefixed with a source tag: `[OP]` for OpenCode, `[CO]` for Copilot, and `[CL]` for Claude Code.

### Searching and Exporting
When a `title` is provided:
1. The tool searches for sessions matching the query across all selected sources.
2. If no sessions are found, an error is printed.
3. If exactly one match is found, it is exported.
4. If multiple matches are found:
   - If `--all` is provided, all are exported.
   - If running in an interactive terminal (`sys.stdin.isatty()`), the user is prompted to choose a specific session by number, 'a' for all, or hit Enter to default to the most recent. The list includes the `[OP]`, `[CO]`, or `[CL]` source tags for clarity.
   - If running non-interactively without `--all`, it defaults to exporting the most recent match.

### Output Locations
- By default, output paths are determined by the `OBSIDIAN_DIR` environment variable.
- **Copilot Default:** `~/Notes/Main/Copilot Chat Archive/`
- **OpenCode Default:** `~/Notes/Main/OpenCode Chat Archive/`
- **Claude Default:** `~/Notes/Main/Claude Chat Archive/`
- Output filenames are generated by sanitizing the session title (removing invalid path characters `<>:"/\|?*`, collapsing whitespace, and truncating to 120 characters) and appending `.md`.

## 2. Low-Level Data Spec: VSCode Copilot

### Workspace Storage
Copilot chat sessions are stored locally per-workspace. The tool iterates through VSCode workspace storage directories located at:
`~/Library/Application Support/Code/User/workspaceStorage/*`

### `state.vscdb` Index Parsing
Each workspace directory contains a `state.vscdb` SQLite database. 
- The index of chat sessions is stored in the `ItemTable`.
- The tool executes: `SELECT value FROM ItemTable WHERE key = 'chat.ChatSessionStore.index'`
- The `value` column contains a JSON string with an `entries` object. The keys of this object are `session_id`s, and the values are metadata objects containing `title`, `lastMessageDate`, and `isEmpty`.

### JSONL Structure
The actual chat history for a session is stored in a JSONL file at `chatSessions/<session_id>.jsonl` within the workspace directory. Each line is a JSON object with a `kind` identifier and a `v` payload.
- **`kind: 0`**: Initial metadata entry. Contains `sessionId`, `creationDate`, and `responderUsername`.
- **`kind: 1`**: Standard turn data or turn completion marker.
  - If `v` is a string, it typically represents the user's message prompt or an auto-generated chat title.
  - If `v` is an object containing `completedAt`, it signifies a turn-completion marker.
- **`kind: 2`**: Response payload. Typically contains an array of token fragments, inline references (e.g., `inlineReference`), HTML rendering data (`supportHtml`), and tool invocations (`invocationMessage` / `pastTenseMessage`). A `kind: 2` entry with an array where the first item contains a `requestId` marks a RESPONSE START.

### Turn Boundaries and Data Extraction
Extracting a clean chat log requires complex parsing to determine turn boundaries:
1. **Response Starts**: The tool identifies "REQID" entries (`kind: 2` with `requestId`) to find where assistant responses begin.
2. **Grouping**: Consecutive REQIDs are grouped into a single turn unless a new user message (a `kind: 1` string longer than 15 characters, excluding titles) appears between them.
3. **User Message**: To find the user message for a turn, the tool looks backward from the first REQID to the previous completion marker. It selects the longest `kind: 1` string as the prompt, with shorter strings assumed to be auto-generated session titles.
4. **Assistant Response**: Content is aggregated from `kind: 2` list items between the first REQID and the completion marker. 
   - **Text**: Standalone text, `supportHtml` fragments, and inline code references are merged and deduplicated.
   - **Tool Calls**: Extracted from `invocationMessage` or `pastTenseMessage` and deduplicated.

## 3. Low-Level Data Spec: OpenCode

### SQLite WAL DB
OpenCode stores its session data globally in an SQLite database at:
`~/.local/share/opencode/opencode.db`
Because OpenCode uses WAL (Write-Ahead Logging) mode, the tool connects using a Read-Only URI (`file:...opencode.db?mode=ro`) to prevent locking issues, falling back to a standard connection if the URI mode fails.

### Table Schemas
The chat state is normalized across several linked tables:

- **`session` Table**:
  - `id`: Unique session identifier.
  - `title`: Display title of the session.
  - `time_created`: Epoch timestamp.
  - `parent_id`: ID of a parent session (for subagent/branched chats).

- **`message` Table**:
  - `id`: Unique message identifier.
  - `session_id`: Foreign key to the `session` table.
  - `time_created`: Epoch timestamp.
  - `data`: A serialized JSON string containing metadata like the `role` ("user" or "assistant").

- **`part` Table**:
  - `message_id`: Foreign key to the `message` table.
  - `time_created`: Epoch timestamp.
  - `data`: A serialized JSON string containing the content payload. Includes a `type` field ("text" or "tool").

### Parent/Child Relationship Parsing
OpenCode sessions can spawn subagents. The script traverses these relationships:
- **Parent**: Queries the `session` table for the row matching the current session's `parent_id`.
- **Children**: Queries the `session` table for rows where `parent_id` matches the current session's ID.
These relationships are explicitly linked in the exported Markdown output using Obsidian wiki-links (e.g., `[[Child Session Name]]`).

### Turn Generation and Assembly
Turns are generated sequentially by a chronological join across tables:
- `SELECT m.data, p.data FROM message m JOIN part p ON m.id = p.message_id WHERE m.session_id = ? ORDER BY m.time_created ASC, p.time_created ASC`
- **Boundary Detection**: A new turn is initiated whenever the `role` parsed from the message data switches from "assistant" to "user" (or for the very first message).
- **Data Aggregation**:
  - If `role` is "user", `text` parts are appended to the user prompt block.
  - If `role` is "assistant" and `type` is "text", the string is appended to the assistant's response block.
  - If `role` is "assistant" and `type` is "tool", the tool name (from `tool`) and its inputs (from `state.input`) are appended to the tool calls block. Tool inputs are serialized to JSON and truncated to 250 characters in the markdown output to maintain readability.

## 4. Low-Level Data Spec: Claude Code

### JSONL Storage
Claude Code chat sessions are stored in JSONL format inside local project directories. The script finds these using the glob pattern:
`~/.claude/projects/*/*.jsonl`
Files smaller than 100 bytes are ignored to skip empty or trivial session files. The session ID is derived directly from the `.jsonl` filename.

### Turn Parsing and Boundaries
The parser reads each line of the JSONL file, ignoring lines where `type` is not `"user"` or `"assistant"`.
- A session's overall `time_created` is extracted by converting the first encountered ISO timestamp (e.g., `2026-03-06T18:41:56.581Z`) to epoch milliseconds. If a timestamp is missing, it is inferred from the file's creation/modification time.
- Turns are accumulated using a boundary check: whenever a new `"user"` message is encountered, and the previous message was from the `"assistant"`, the current turn is finalized, and a new turn begins.
- Sessions with 0 turns are no longer strictly filtered out. If a session contains only `file-history-snapshot` events (and thus no user messages), it is included as an "Untitled Claude Session".

### User Messages & Title Extraction
When `role` is `"user"`:
- The content can be a string or a list. If it is a list where the first item is `{"type": "tool_result"}`, the message is treated as a tool execution result and skipped from the main user text flow.
- Regular user content is extracted from objects with `{"type": "text"}` and joined into a single string.
- The session title is derived from the first user message. To prevent ugly titles, `clean_user_text()` strips out metadata XML blocks like `<ide_selection>`, `<ide_opened_file>`, `<local-command-caveat>`, `<command-name>`, `<command-message>`, `<command-args>`, and `<local-command-stdout>`.
- Additionally, if the JSON data indicates `"isMeta": true` (usually IDE context or background messages), the text is excluded from the session title but appended to the turn output as `_Meta: {text}_` for context.

### Assistant Responses & Tool Calls
When `role` is `"assistant"`:
- The content is a list of items.
- Items with `{"type": "text"}` are appended to the assistant's textual response.
- Items with `{"type": "tool_use"}` indicate tool executions. The parser extracts the `name` of the tool and its associated `input` dictionary, formatting it nicely in a markdown block quote alongside the assistant text. Tool inputs are truncated at 250 characters for display to prevent overly noisy outputs.
